# Data Map — MedDefense Health Systems

## Data Category Table

| Data Category | Classification | At Rest | In Transit | In Use | Current Protection | Protection Gaps |
|---|---|---|---|---|---|---|
| Patient medical records | Restricted | `ehr-db-01` | Flat internal network, no segmentation | Nurse stations, physician workstations | SSH key-auth on `ehr-srv-01` only; nightly backup; password policy | DB reachable from entire /16, not just app server; nurse station session left unlocked and visible (Task 3, Obs. 3); no MFA; no DLP to catch exfiltration |
| Medical imaging data | Restricted | `pacs-srv-01` | Flat internal network | Radiology workstations, shared login | Basic network position only | Not covered by any backup (Task 4/5); shared credential `raduser/radiology1` removes individual accountability |
| Real-time clinical/vitals data | Restricted | Transient on device (monitors, pumps) | Flat network, shared with general workstations | Bedside display, nurse monitoring | Perimeter firewall only (no device-level control) | No VLAN isolation for medical IoT; BD Alaris firmware has an unmitigated known CVE; `MON-VITALS-3F-01` vendor still unconfirmed |
| Financial/billing data | Confidential | `billing-srv-01` (MySQL) | Flat internal network, port 3306 exposed org-wide | Finance workstations | Nightly backup | MySQL reachable from the entire internal network, not restricted; server already compromised twice (ransomware, cryptominer) via an unpatched Apache |
| Employee HR records | Confidential | `file-srv-01` | Flat internal network | HR department workstations | Standard file share permissions, password policy | An unmanaged personal laptop had 3 weeks of network reachability to this exact share (Task 1, Incident F) — a proven, not theoretical, gap |
| System credentials | Restricted | AD (`ad-dc-01/02`); `sshd_config` on servers; a laminated sheet on the network closet wall | Authentication traffic on the flat network | Login prompts, admin sessions | Password complexity policy, account lockout, key-only SSH on `ehr-srv-01` only | Switch management credentials are physically taped to a wall (Task 3, Obs. 2); no MFA anywhere in the environment; most Linux servers still allow password-based SSH |
| Audit/security logs | Confidential | Scattered locally per device (firewall, Event Viewer, `/var/log`, Apache logrotate) | Not forwarded anywhere | Reviewed manually only "when something breaks" | Local retention only (30 days firewall, 4 weeks Apache) | No centralization, no log integrity protection, no automated alerting — this is why the cryptominer ran undetected for two weeks (Task 2) |
| Research/clinical trial data | Restricted | Dr. Patel's personal NAS (unmanaged, Task 11) | Unknown — personal device on office wall port | Dr. Patel's own access; scope unclear | None — entirely outside corporate governance | Completely invisible to every control in the Task 10 matrix; no backup, no access policy, no monitoring |

## Data Risk Summary

MedDefense's most significant data protection weakness is **system credentials at rest and in use**. Credentials are the classification level "Restricted" by definition — they unlock access to every other category in this table — yet the network closet's switch management password is physically written on paper taped to a wall, and no system anywhere in the environment requires a second authentication factor. This single combination (trivially exploitable, unlimited downstream reach) is a wider gap relative to its classification than any other row: it doesn't just expose one data category, it removes the last line of defense for all of them at once.
