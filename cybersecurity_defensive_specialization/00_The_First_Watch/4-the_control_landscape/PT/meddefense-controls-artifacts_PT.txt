ARTEFATO 1: TRECHO DE CONFIGURAÇÃO DE FIREWALL (FortiGate 100F)
========================================================================

Fonte: Sarah Park (exportado do console de gerenciamento do FortiGate)
Nota da Sarah: "Esta é uma exportação parcial. A config completa tem mais
de 2000 linhas. Eu extraí as regras de política relevantes."

config firewall policy

    edit 1
        set name "Allow-Web-Inbound"
        set srcintf "wan1"
        set dstintf "dmz"
        set srcaddr "all"
        set dstaddr "web-srv-01"
        set action accept
        set service "HTTP" "HTTPS"
        set logtraffic all
        set status enable
    next

    edit 2
        set name "Allow-VPN-Westside"
        set srcintf "vpn-westside"
        set dstintf "internal"
        set srcaddr "westside-subnet"
        set dstaddr "server-subnet"
        set action accept
        set service "ALL"
        set logtraffic utm
        set status enable
    next

    edit 3
        set name "Allow-VPN-HQ"
        set srcintf "vpn-hq"
        set dstintf "internal"
        set srcaddr "hq-subnet"
        set dstaddr "server-subnet"
        set action accept
        set service "ALL"
        set logtraffic utm
        set status enable
    next

    edit 4
        set name "Allow-Internal-Outbound"
        set srcintf "internal"
        set dstintf "wan1"
        set srcaddr "all"
        set dstaddr "all"
        set action accept
        set service "ALL"
        set nat enable
        set logtraffic utm
        set status enable
    next

    edit 5
        set name "Deny-All"
        set srcintf "any"
        set dstintf "any"
        set srcaddr "all"
        set dstaddr "all"
        set action deny
        set logtraffic all
        set status enable
    next

end

Nota do Marcus (encontrada em suas anotações):
"As regras 2 e 3 permitem TODOS os serviços vindos da VPN. Isso é
permissivo demais. Se o roteador consumidor da Westside for
comprometido, o atacante tem acesso total à subnet de servidores.
Precisa ser restrito a portas específicas."

"A regra 4 permite TODA saída. Não há filtragem de egress. O minerador
de criptomoeda no billing-srv-01 está conversando tranquilamente com
pools de mineração porque nada bloqueia tráfego de saída para portas
arbitrárias."

ARTEFATO 3: POLÍTICA DE SENHAS (Documento Interno -- arquivo Word)
========================================================================

Fonte: Drive compartilhado do RH

             MEDDEFENSE HEALTH SYSTEMS
             Política de Segurança da Informação: Requisitos de Senha
             Versão: 1.2 | Aprovado: [2 anos atrás] | Autor: Departamento de TI

1. ESCOPO
   Esta política se aplica a todos os funcionários, contratados e
   fornecedores da MedDefense que acessam os sistemas de informação
   da MedDefense.

2. REQUISITOS DE SENHA
   - Tamanho mínimo: 8 caracteres
   - Complexidade: deve conter ao menos uma letra maiúscula, uma
     minúscula, um número e um caractere especial
   - Rotação: as senhas devem ser trocadas a cada 90 dias
   - Histórico: o sistema lembra as últimas 5 senhas
   - Bloqueio: contas são bloqueadas após 5 tentativas falhas, por 30 minutos

3. CONTAS COMPARTILHADAS
   Contas compartilhadas são desencorajadas, mas podem ser usadas quando
   contas individuais não são tecnicamente viáveis. As senhas de contas
   compartilhadas devem ser trocadas quando qualquer usuário com acesso
   deixa a organização.

4. AUTENTICAÇÃO MULTIFATOR
   MFA é recomendado para acesso remoto, mas não é atualmente obrigatório.

5. APLICAÇÃO
   Os requisitos de senha são aplicados através de Group Policy do Active
   Directory para sistemas Windows. Sistemas Linux são configurados
   individualmente.

   Aprovado por: Sarah Park, Diretora de TI
   Última revisão: [18 meses atrás]

ARTEFATO 8: GESTÃO DE LOGS (do Tom Reeves, resumo verbal)
========================================================================

"O que fazemos com os logs? Bom...

- O FortiGate guarda os próprios logs localmente. Retenção de 30 dias.
  Não encaminhamos para lugar nenhum.
- Servidores Windows enviam eventos para o Event Viewer. Verificamos
  manualmente quando algo quebra.
- Servidores Linux escrevem em /var/log. Syslog padrão. Sem centralização.
- Logs do Apache no web-srv-01 e no billing-srv-01 giram semanalmente
  via logrotate. Mantemos 4 semanas.
- A aplicação de EHR tem seu próprio log de auditoria. O fornecedor
  gerencia isso. Podemos solicitar exportações, mas leva 48 horas.
- O Active Directory registra eventos críticos, mas não temos alertas.
  Só olhamos os logs do AD quando algo quebra.

O Marcus ficava falando de um SIEM. Uma coisa chamada 'Wazuh'. Ele
começou a pesquisar mas nunca chegou a instalar nada. Acho que ele
tinha umas anotações sobre isso em algum lugar."

Não existe sistema centralizado de gestão de logs.
Não existe alerta automatizado sobre eventos de segurança.
Não existe proteção de integridade dos logs (sem hashing, sem
armazenamento write-once).
