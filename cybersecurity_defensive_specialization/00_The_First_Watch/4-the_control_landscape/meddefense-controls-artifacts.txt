ARTIFACT 1: FIREWALL CONFIGURATION EXTRACT (FortiGate 100F)
========================================================================

Source: Sarah Park (exported from FortiGate management console)
Note from Sarah: "This is a partial export. The full config is 2000+ lines.
I pulled the relevant policy rules."

config firewall policy

    edit 1
        set name "Allow-Web-Inbound"
        set srcintf "wan1"
        set dstintf "dmz"
        set srcaddr "all"
        set dstaddr "web-srv-01"
        set action accept
        set service "HTTP" "HTTPS"
        set logtraffic all
        set status enable
    next

    edit 2
        set name "Allow-VPN-Westside"
        set srcintf "vpn-westside"
        set dstintf "internal"
        set srcaddr "westside-subnet"
        set dstaddr "server-subnet"
        set action accept
        set service "ALL"
        set logtraffic utm
        set status enable
    next

    edit 3
        set name "Allow-VPN-HQ"
        set srcintf "vpn-hq"
        set dstintf "internal"
        set srcaddr "hq-subnet"
        set dstaddr "server-subnet"
        set action accept
        set service "ALL"
        set logtraffic utm
        set status enable
    next

    edit 4
        set name "Allow-Internal-Outbound"
        set srcintf "internal"
        set dstintf "wan1"
        set srcaddr "all"
        set dstaddr "all"
        set action accept
        set service "ALL"
        set nat enable
        set logtraffic utm
        set status enable
    next

    edit 5
        set name "Deny-All"
        set srcintf "any"
        set dstintf "any"
        set srcaddr "all"
        set dstaddr "all"
        set action deny
        set logtraffic all
        set status enable
    next

end

Note from Marcus (found in his notes):
"Rules 2 and 3 allow ALL services from VPN. This is too permissive.
If Westside's consumer router gets compromised, the attacker has full
access to the server subnet. Needs to be locked down to specific ports."

"Rule 4 allows ALL outbound. No egress filtering. The crypto-miner on
billing-srv-01 is happily talking to mining pools because nothing stops
outbound traffic to arbitrary ports."

ARTIFACT 3: PASSWORD POLICY (Internal Document -- Word file)
========================================================================

Source: HR Department shared drive

             MEDDEFENSE HEALTH SYSTEMS
             Information Security Policy: Password Requirements
             Version: 1.2 | Approved: [2 years ago] | Author: IT Department

1. SCOPE
   This policy applies to all MedDefense employees, contractors and
   vendors who access MedDefense information systems.

2. PASSWORD REQUIREMENTS
   - Minimum length: 8 characters
   - Complexity: must contain at least one uppercase letter, one
     lowercase letter, one number and one special character
   - Rotation: passwords must be changed every 90 days
   - History: the system remembers the last 5 passwords
   - Lockout: accounts lock after 5 failed attempts for 30 minutes

3. SHARED ACCOUNTS
   Shared accounts are discouraged but may be used when individual
   accounts are not technically feasible. Shared account passwords
   must be changed when any user with access leaves the organization.

4. MULTI-FACTOR AUTHENTICATION
   MFA is recommended for remote access but is not currently required.

5. ENFORCEMENT
   Password requirements are enforced through Active Directory Group
   Policy for Windows systems. Linux systems are configured individually.

   Approved by: Sarah Park, IT Director
   Last reviewed: [18 months ago]

ARTIFACT 8: LOG MANAGEMENT (from Tom Reeves, verbal summary)
========================================================================

"What do we do with logs? Well...

- The FortiGate stores its own logs locally. 30 days retention. We don't
  forward them anywhere.
- Windows servers send events to the Event Viewer. We check manually if
  something breaks.
- Linux servers write to /var/log. Standard syslog. No centralization.
- Apache logs on web-srv-01 and billing-srv-01 rotate weekly via logrotate.
  We keep 4 weeks.
- The EHR application has its own audit log. The vendor manages that.
  We can request exports but it takes 48 hours.
- Active Directory logs critical events but we have no alerting.
  The only time we look at AD logs is when something breaks.

Marcus kept talking about a SIEM. Something called 'Wazuh.' He started
researching it but never got to install anything. I think he had some
notes on it somewhere."

No centralized log management system exists.
No automated alerting on security events.
No log integrity protection (no hashing, no write-once storage).
