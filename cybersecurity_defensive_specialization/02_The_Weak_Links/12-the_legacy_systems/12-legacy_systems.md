# 12. The Legacy Systems — MedDefense Vulnerability Scan

**Analyst:** Threat Intelligence Analyst
**Date:** Current

## System 1: Windows XP SP3 (`WS-RAD-01`, 10.10.1.70, MRI Workstation)

**EOL Research:** Searching NVD and vendor/researcher sources for new Windows XP CVEs published in the last 2 years returns effectively **zero** new disclosures naming Windows XP specifically. This is not good news — it means the CVE ecosystem itself has stopped watching this OS, not that it has become safe. The "2 most critical" vulnerabilities relevant to this host are still the ones already in the scan report from years past: **CVE-2008-4250 (MS08-067, CVSS 10.0)** and **CVE-2019-0708 (BlueKeep, CVSS 9.8)** — both wormable, both already confirmed present. No newer research exists because no vendor or CNA is still looking.

**Permanent Exposure:** "Unpatched" describes a vulnerability with a known fix that has not yet been applied — a temporary, closeable gap. "EOL" describes an OS for which **no fix will ever be produced again**, for any future vulnerability, no matter how severe. Every new SMB, RDP or kernel-level flaw disclosed against contemporary Windows from this point forward is a flaw MedDefense can never assume is absent from this machine, because no one will ever confirm or deny it, let alone patch it — the exposure grows every year with zero possibility of closure through patching.

**Scan Findings:** Only Finding 004, but it bundles three separately weaponized CVEs (MS08-067, EternalBlue/CVE-2017-0144, BlueKeep/CVE-2019-0708) on ports 445 and 3389. All three are exploitable **specifically because** the OS is EOL — none has a vendor patch available for Windows XP, and the finding exists purely because this device can never receive the fixes that closed these same holes on every supported OS a decade ago.

**Compensating Controls:** The 1x00 Task 6 strategy proposed three controls: **Control 1** (network segmentation — isolate the MRI onto its own VLAN, firewalled to communicate only with the PACS server), **Control 2** (a formal, documented risk-acceptance and quarterly-review process), and **Control 3** (physical access restriction and USB port lockdown). **These controls do not adequately address the scan findings** — because they were never actually implemented. The 1x01 Task 7 Attack Surface Map already noted the MRI's compensating controls as "still only proposed, not implemented," and this scan proves it directly: Finding 004 shows ports 445 and 3389 reachable from the general workstation subnet, which is exactly the exposure Control 1 was designed to close. Had it been deployed, this finding would show a device unreachable from anywhere except PACS, not open to the entire 10.10.1.0/24 range. **Additional recommendation:** even after Control 1 is finally implemented, add a host-based or segment-level IDS/IPS rule specifically watching for SMB/RDP exploitation signatures on the MRI's microsegment, since GAP-002 (no detection anywhere) means even a successfully-isolated MRI would still be compromised silently if the one remaining permitted channel (PACS) were ever itself compromised.

## System 2: Windows Server 2012 R2 (`print-srv-01`, 10.10.2.31, Print Server)

**EOL Research:** Unlike Windows XP, Server 2012 R2 is recent enough to still receive research attention and Extended Security Updates (ESU, available through October 13, 2026), so NVD searches return real, current disclosures. The two most critical found: **CVE-2024-49112 ("LDAPNightmare," CVSS 9.8)** — a zero-click, unauthenticated integer-overflow RCE in Windows LDAP, with 10+ public PoCs on GitHub — and **CVE-2025-24045 (Windows RDS RCE, CVSS 8.1)** — an unauthenticated network RCE via a race condition in Remote Desktop Services, both explicitly listed by Microsoft as affecting Server 2012 R2.
**Permanent Exposure:** MedDefense's `print-srv-01` is not enrolled in ESU, meaning it is already past its own effective patch cutoff even though ESU technically still exists for organizations that pay for it. As with Windows XP, every future CVE affecting these same OS components (LDAP client libraries, RDS, the Print Spooler service already exploited via PrintNightmare) accumulates permanently — the difference from Windows XP is only a matter of degree and time, not kind.

**Scan Findings:** Finding 008 (Windows Server 2012 R2 EOL, bundling CVE-2021-34527/PrintNightmare, CVSS 8.8, with the Print Spooler service actively running). This is exploitable specifically because the OS is EOL in the sense that matters here: MedDefense could patch PrintNightmare itself if actively supported, but the same lack of ESU enrollment that leaves this CVE open also leaves every other Server 2012 R2 CVE disclosed since (including the two found above) permanently open on this host as well.

**Compensating Controls:** No compensating control strategy for `print-srv-01` was proposed anywhere in 1x00 — Task 6 addressed only the MRI. GAP-013 (1x00 Task 12) rated this specific asset's risk as **Low**, citing internal-network-only exposure as a partial compensating factor and explicitly agreeing with a prior internal assessment (Marcus's own M-08 note) that this is "a compliance issue more than an active exploitation risk." **Recommendation:** at minimum, restrict which subnets can reach ports 135/139/445/9100 on this host via host or network firewall rules (mirroring the segmentation logic already proposed for the MRI), since print services have no legitimate reason to be reachable from the medical-device or server subnets.

## System 3: Ubuntu 18.04 LTS without ESM (`billing-srv-01`, 10.10.2.15, Billing Server)

**EOL Research:** Because Ubuntu 18.04's exact package versions (Apache 2.4.29 among them) are frozen in time without ESM, any CVE disclosed against Apache HTTP Server up to 2.4.59/2.4.60 applies directly and permanently to this host. Two confirmed, highly relevant, recent examples: **CVE-2024-38474 (CVSS 9.8)** — an improper-escaping flaw in `mod_rewrite` allowing script execution outside intended directories or source disclosure, affecting Apache 2.4.59 and earlier (2.4.29 is well within range) — and **CVE-2024-38475 (CVSS 9.1)** — a related `mod_rewrite` output-escaping flaw in the same version range enabling URL-to-filesystem mapping abuse, confirmed by Canonical's own security tracker as applicable to Ubuntu 18.04 "bionic." Both were disclosed in **July 2024** — meaning this exact host has accumulated two more unpatchable Critical vulnerabilities in the same Apache component already responsible for two real prior compromises, in just the last two years.

**Permanent Exposure:** Ubuntu 18.04 without ESM is functionally identical to Windows XP in kind, just younger in years — standard security support ended June 2023, and without an Ubuntu Pro/ESM subscription, no patch will ever arrive for any newly disclosed CVE in any installed package (kernel, Apache, OpenSSH, glibc), forever. The only difference between this host and Windows XP is how many years of accumulated unpatched CVEs currently exist — and that number grows identically over time either way.

**Scan Findings:** Findings 001 (CVE-2021-44790), 002 (CVE-2019-0211), 006 (MySQL misconfiguration, not EOL-related), 009 (SSH misconfiguration, not EOL-related), 011 (the EOL/ESM status itself), and 026 (47 unpatched kernel CVEs). Findings 001, 002, and 026 are exploitable **specifically because** the OS is EOL — patches for all three exist upstream and simply never reach this host without ESM enrollment.

**Compensating Controls:** No compensating control strategy exists for `billing-srv-01` anywhere in 1x00 — it was never treated as a "legacy dilemma" the way the MRI was, despite having already been compromised twice. **Recommendation:** enroll in Ubuntu Pro/ESM immediately as the lowest-cost, highest-leverage compensating control available (a subscription, not a migration project), combined with the network-restriction fixes already recommended for Findings 006/009 in earlier tasks and a host-based firewall restricting inbound access to ports 80/3306/22 to only the specific hosts that require them.

## Business Decision: Which System Migrates First?

**Recommendation: `billing-srv-01` (Ubuntu 18.04) migrates first.**

By raw asset criticality alone (1x00 Task 8), the MRI workstation ranks highest — Medical IoT is rated Overall Critical, driven by direct patient-safety impact, while Billing & Financial Infrastructure is rated only Overall **High**, and `print-srv-01` specifically was already assessed as Low risk in GAP-013. If criticality were the only input, the MRI would win outright.

But "migrate one system off EOL in one quarter" is not the same question as "which asset matters most in the abstract." The MRI's replacement is not a software upgrade — it is a capital medical-device procurement, almost certainly requiring vendor validation and regulatory recertification tied to the specific hardware/OS bundle (which is precisely why 1x00 Task 6 already concluded it "cannot be patched, upgraded, replaced or disconnected" and proposed *network segmentation*, not migration, as the answer). A single quarter's budget cannot realistically replace a certified imaging device; the correct quarter-one action for the MRI is finally **implementing** Control 1, not migrating it. `print-srv-01`, meanwhile, is both low-criticality and low-exposure (GAP-013) — fixing it first would be the least impactful use of a single quarter's budget of the three.

`billing-srv-01` is the one system where "migrate" is both operationally realistic (an OS upgrade or an ESM subscription, not a capital device replacement) and where the threat exposure is not theoretical: this is the only asset among the three with **confirmed, repeated, real-world exploitation** (ransomware, then a cryptominer, both via this exact Apache instance, per 1x00 Tasks 1-2), it is **internet-facing** (1x01 Task 6, Kill Chain #4 Step 1) rather than internal-only, and — as this task's own EOL research shows — it is actively accumulating brand-new Critical CVEs (CVE-2024-38474, CVE-2024-38475) in the same vulnerable component right now, not just carrying historical debt. Migrating `billing-srv-01` off EOL this quarter closes the highest-likelihood, most actively-exploited, most operationally-achievable gap of the three — while the MRI's real quarter-one fix (segmentation) and the print server's lower-cost network restriction can proceed in parallel without competing for the same capital-project budget.
